# Validate And Run Your First Workflow

This tutorial gets you from a fresh checkout to one validated workflow and one submitted job.

## What You Will Do

By the end, you will have:

Launch a complex AI workflow with one command, and the runtime automatically resolves the required configuration. Once running, the workflow depends only on local resources, including locally available AI models. It can continue operating without internet connectivity.
## Before You Start

You need:

- A PC with at least 16GB RAM running macOS, Linux, or Windows with WSL2
- [Docker](https://www.docker.com/products/docker-desktop) installed and running

If those are not ready yet, follow [Installation](installation.md) first.

## Step 1: install the runtime

```
curl -fsSL https://mirrorneuron.io/install.sh | bash
```

This will install the `mn` CLI and the runtime and start it automatically. If you already have the runtime installed, you can start it with:

```bash
mn runtime start
```

This checks the local manifest and input contract without submitting a runtime job.

## Step 2: Start a job from a blueprint

```bash
mn blueprint run vc_assistant
```

This is a simple workflow to act like a VC scout to analysis a few startup, using both web search and a local LLM model. The CLI will print the job id and live events and numerical tools to quick screen their value. The method we are using is from common VC practices. 
You can close the winodw, the job will continue to run in the background. You can check the job status with: 

```bash
mn job monitor <job_id>
```

## Step 3: Check the result

Typically, the job will running in more than 10 minutes, depends on yout PC. Once it's completed. You can have its results at a local folder:

```bash
~/Downloads/vc_assistant
```

## Step 4: Run The Workflow

```bash
mn blueprint run message_routing_trace
```

Expected output:

```text
Job submitted successfully
```

The CLI may also print live events and the job id. Keep the job id for inspection commands.

## Step 5: Inspect Jobs

```bash
mn job list
```

Expected output:

```text
Job ID
```

Check a single job:

```bash
mn job status <job_id>
```

Expected output includes:

```json
{
  "status": "completed"
}
```

If the job is still running, wait a moment and run `mn job status <job_id>` again.

## Step 6: Try A Python-Defined Blueprint

Run the checked-in Python SDK research pipeline:

```bash
mn blueprint run python_sdk_research_pipeline
```

Expected output:

```text
Job submitted successfully
```

For Python bundle-generation details, see [Python SDK](SDK.md).

## Security Basics

Before running bigger or third-party workflows:

- Review `manifest.json` and `payloads/`.
- Check whether a node uses `host_local` or OpenShell.
- Check `pass_env` before secrets are exposed to workers.
- Use dry-run options for email, Slack, or other external delivery flows.
- Treat live messages and model outputs as untrusted input.

Read [Security Model](security.md) before exposing a runtime to other users or machines.

## Troubleshooting

### `mn: command not found`

Install the CLI or activate the project virtual environment. Then verify:

```bash
which mn
mn --help
```

### gRPC connection refused

The runtime is not reachable.

```bash
mn runtime start
mn node list
```

### Redis connection errors

Check:

```bash
docker exec mirror-neuron-redis redis-cli ping
```

Expected output:

```text
PONG
```
