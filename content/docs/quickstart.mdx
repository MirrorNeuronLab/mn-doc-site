# Run Your First Workflow

This guide validates a checked-in blueprint, starts the local runtime, submits a workflow, and shows where to inspect the result.

## Before You Start

You need Docker and the `mn` CLI. Complete [Installation](installation.md) first. Blueprint requirements vary: a blueprint may need input files, model access, OpenShell, or other local services. Review its `README.md`, `manifest.json`, and `pass_env` before running it.

## 1. Validate a Local Blueprint

From the workspace root:

```bash
mn blueprint validate otterdesk-blueprints/generic_customer_service_voice_coworker
```

Expected output includes a successful validation result. This preflight does not submit a job.

## 2. Start and Check the Runtime

```bash
mn runtime start
mn runtime health
mn node list
```

`mn runtime health` confirms local runtime dependencies; `mn node list` shows the local node or any configured cluster nodes.

## 3. Submit the Workflow

```bash
mn blueprint run --folder otterdesk-blueprints/generic_customer_service_voice_coworker
```

The command prints a job ID and may print a run ID. Keep both values: the job ID is for runtime control and the run ID identifies the local run store.

## 4. Inspect the Job

```bash
mn job list
mn job status <job_id>
mn job monitor <job_id>
```

Use `mn job cancel <job_id>` to stop a running workflow. For blueprint-run artifacts and logs, use:

```bash
mn blueprint logs <run_id>
mn blueprint tail <run_id>
mn blueprint export <run_id> --format markdown
```

By default, run records are stored under `~/.mn/runs/<run_id>/`. Set `MN_HOME` or `MN_RUNS_ROOT` to use another location.

## 5. Explore Catalog Blueprints

List catalog blueprints configured for the runtime:

```bash
mn blueprint list
```

Run a catalog blueprint by its ID only after reviewing its inputs and required services:

```bash
mn blueprint run <blueprint_id>
```

For local checkout examples, use [Examples](examples.md).

## Troubleshooting

If `mn` is unavailable, confirm it is on your `PATH`:

```bash
which mn
mn --help
```

If the runtime is unreachable:

```bash
mn runtime start
mn runtime health
```

For model preflight failures, inspect the requested model:

```bash
mn model list
mn model doctor gemma4:e2b
```

Continue with [Troubleshooting](troubleshooting.md) for Redis, OpenShell, Docker, and cluster failures.
