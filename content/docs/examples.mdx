# Choose a Blueprint Example

The checked-in `otterdesk-blueprints` directory contains task-focused workflows. They are not interchangeable smoke tests: each has its own inputs, model needs, services, and safety limits. Read the blueprint's `README.md` and `SPEC.md` before launching it.

## Available Local Examples

| Blueprint folder | Purpose |
| --- | --- |
| `drug_discovery_research_assistant` | Produces evidence-backed early drug-discovery research and candidate summaries. |
| `financial_advisor` | Prepares review-only household-finance, tax, and portfolio reports from supplied documents. |
| `generic_customer_service_voice_coworker` | Starts a local voice customer-service experience with editable business knowledge. |
| `medical_deid_record_intake_assistant` | Detects PII/PHI risks and prepares a review-gated de-identification packet. |
| `personal_legal_assistant` | Reviews invoices, bills, and contracts against supplied legal-playbook context. |
| `property_deal_research_assistant` | Researches and compares real-estate opportunities using supplied constraints. |
| `safety_video_analyser` | Analyzes local workplace footage and writes a structured safety report. |
| `vc_assistant` | Produces reviewable startup-screening reports from supplied documents. |
| `video_watch_assistant` | Monitors an approved local or mapped video stream and writes observation artifacts. |

## Recommended First Steps

Start with the blueprint whose input domain you understand. Validate the local folder before trying a full run:

```bash
mn blueprint validate otterdesk-blueprints/<blueprint_folder>
```

Then start the runtime and submit it:

```bash
mn runtime start
mn blueprint run --folder otterdesk-blueprints/<blueprint_folder>
```

Inspect the returned job ID and run ID with:

```bash
mn job status <job_id>
mn blueprint logs <run_id>
```

## Catalog Runs

When your runtime has a catalog configured, list its IDs with:

```bash
mn blueprint list
```

Then run one by ID:

```bash
mn blueprint run <blueprint_id>
```

The catalog can be a Git repository or a local directory; see [Environment Variables](env_variables.md) for `MN_BLUEPRINT_SOURCE`, `MN_BLUEPRINT_REPO`, and `MN_BLUEPRINT_LOCAL`.

## Safety Notes

- Review `manifest.json`, payloads, and `pass_env` before granting access to files, networks, or secrets.
- Use sample, mock, quick-test, or dry-run settings when the blueprint offers them.
- Treat finance, tax, legal, medical, safety, and model outputs as review material, not autonomous decisions.
- Cancel long-running or service workflows when you are done: `mn job cancel <job_id>`.
