# Documentation Style

Write pages that are clear, practical, and backed by the implementation currently in the workspace.

## Page Types

| Type | Purpose |
| --- | --- |
| Tutorial | Walk a reader through a complete, safe first path. |
| How-to | Solve one operational task. |
| Reference | Give exact commands, configuration, or API facts. |
| Explanation | Describe architecture and tradeoffs. |

## Commands and Examples

- Use fenced `bash` blocks for commands and state the working directory when it matters.
- Use blueprint folders that exist in `otterdesk-blueprints`; never publish synthetic or deleted example IDs.
- Show a stable success marker only when it is verified. Otherwise say what the reader should inspect.
- Use placeholders such as `<blueprint_id>` when the command depends on local catalog configuration.

For example:

```bash
mn blueprint validate otterdesk-blueprints/<blueprint_folder>
```

## Accuracy and Safety

- Update documentation in the same change as user-visible behavior, commands, ports, configuration names, or API routes.
- Call out security, data-loss, network-exposure, and irreversible-cleanup implications.
- Do not publish agent instructions, release procedures, templates, obsolete diagrams, or references to removed components.
- Link only to pages that exist in this documentation site.

## Review Checklist

- [ ] The page has one clear purpose.
- [ ] Commands reflect the current CLI or component implementation.
- [ ] Examples use existing blueprint folders or explicit placeholders.
- [ ] Local links resolve.
- [ ] Security implications are described where relevant.
